Auto, or open the app?

A Mumbai ride-fare decision tool that's honest about when it actually knows the answer.

Mumbai's auto-rickshaw meter is government-fixed and published — exact, surge-proof, no guessing. Uber/Ola/Rapido pricing is proprietary and dynamic — it can only ever be estimated. Most comparison tools blur that line and give you a confident-looking number either way. This one doesn't: a strong verdict when the gap is wide, a plain "toss-up" when it isn't — with the surge-proof meter as your anchor exactly when app prices are least trustworthy.

How it works

Three steps, two of them deterministic.

1

Exact meter math

₹26 minimum (first 1.5km) + ₹17.14/km, +25% night surcharge — the actual Maharashtra RTO tariff, computed exactly. No model, no guess.

2

Modeled app fares

Uber/Ola/Rapido don't publish rate cards, so app-auto and app-car fares are estimated from typical observed pricing — and labeled as estimates, not facts.

3

Confidence-calibrated verdict

The gap between meter and cheapest app option decides confidence — not the rupee amount. Small gap = "toss-up," not a false-sure recommendation.

Tech stack

Each layer does one job — the LLM is never the one doing arithmetic.

Fare Engine

Deterministic Python

Official RTO meter formula + modeled app rate cards (auto + car tiers). Every fare is reproducible and grep-able — never an LLM guess.

Decision Logic

Confidence engine

Turns fare numbers into a verdict: gap-based confidence, surge-proof anchoring, no-auto-zone handling, and a car-tier "don't default to the cab" call-out.

NL Assistant

Claude (Haiku)

Parses messy questions into structured inputs, and turns the verdict back into plain language. Never computes a fare — structurally can't.

API

FastAPI

/routes, /decide, /ask — deployed on Render. The dropdown demo below hits /decide directly, no LLM cost.

Evals

How the tool's confidence calibration was tested.

A

Decision engine: 22/22

Verdict + confidence on 22 route/time/surge combinations checked against an independently re-derived RTO formula and the documented confidence thresholds — not just tested against its own code.

B

High-confidence precision: 100%

Primary metric. Of every route the NL assistant matched with confidence, 15/15 were correct (live Claude calls, not cached). A wrong high-confidence match is the failure mode this whole project exists to prevent.

C

Correct refusal: 6/6

Out-of-scope queries (wrong city, gibberish, an unsupported South Mumbai pair) were all correctly declined instead of guessing a plausible-sounding wrong route.

Honest caveat: 21 NL test cases is enough to validate the design (confidence-gated refusal works, route matching handles varied phrasing), not enough to claim the assistant is bulletproof. Harder adversarial cases — typos, two known locations in one ambiguous query — are the natural next eval to add. Full per-case results in evals/results.md.
